## Die Bausteine {#bausteine}

| Schicht | Technologie | Wofür |
|---|---|---|
| Frontend | Next.js 16 + React Server Components | Server-rendert das UI, streamt es per HTTP an den Browser |
| UI-Bibliothek | Tailwind + shadcn/ui + Fumadocs | Designsystem, Komponenten, Hilfe-Center |
| Backend | TypeScript + Drizzle ORM | Typsichere Datenbank-Queries direkt aus dem App-Code |
| Datenbank | PostgreSQL (Supabase-Hosting) | Pubs, Personen, Analyse-Ergebnisse, Press-Releases |
| Vektor-Storage | pgvector | 768-dim SPECTER2-Embeddings pro Pub |
| LLM-Layer | OpenRouter (BYOK) | Inhaltliche Pub-Bewertung über Claude, GPT, DeepSeek |
| Embedding | SPECTER2 (Allen AI) | Semantische Press-Similarity, lokal lauffähig |
| ETL | TYPO3-MySQL → Postgres | WebDB-Import als UPSERT pro `webdb_uid` |

## Drizzle ORM: typsicher statt Roh-SQL {#drizzle}

Story Scout greift seit der Migration im Mai 2026 mit **Drizzle ORM** auf
die Datenbank zu. Vorher: rohe SQL-Strings, in TypeScript-Templates
eingebettet. Risiko: Spaltennamen-Typos oder Schema-Drift werden erst
zur Laufzeit entdeckt, typischerweise durch einen 500-Fehler in der UI.

Mit Drizzle wird das Datenbank-Schema im TypeScript-Code abgebildet
(`lib/server/db/schema.ts`). Queries lesen sich wie normaler TS-Code:

```ts
import { eq } from 'drizzle-orm';
import { db, publications } from '@/lib/server/db';

const pub = await db.query.publications.findFirst({
  where: eq(publications.id, pubId),
  with: { pressRelease: true },
});
```

Der TypeScript-Compiler kennt das Schema und meldet bei `publications.titel`
(statt `publications.title`) sofort einen Fehler, Stunden vor dem Deploy.

<Callout type="info">
**Effekt für End-User:** weniger 500-Fehler durch Tipper im SQL,
schnellere Refactorings (Spaltenumbenennungen finden alle Call-Sites
beim Build), und sauberere Lade- und Fehlerseiten, weil weniger
Edge-Cases überhaupt entstehen.
</Callout>

## Migrations & Schema-Pflege {#migrations}

Schema-Änderungen leben in `supabase/migrations/` als nummerierte
SQL-Dateien (`20260514000001_*.sql`). Beim Deploy spielt Supabase die
neuen Migrations automatisch ein. Jede Migration ist **idempotent**, also
neu-anwendbar ohne Schaden. Wenn die Spalte schon existiert, passiert
nichts.

Architecture Decision Records (ADRs) in `docs/adr/` halten fest, warum
bestimmte Entscheidungen so getroffen wurden, etwa „warum keine ISR"
(ADR 0009) oder „warum repository-layer über raw queries" (ADR 0008).

## RSC vs. Client Components {#rsc}

Next.js 16 lässt jede Komponente entweder **serverseitig** (RSC,
Default) oder **clientseitig** (`'use client'`) rendern. Story Scout
nutzt RSC, wo immer möglich:

- **Server**: Datenladen, Dashboards, Pub-Listen, Hilfe-Artikel
- **Client**: Filter-Dialoge, Triage-Buttons, MeisterTask-Push, Charts

Das spart Client-Bundle-Größe und macht Initial-Loads spürbar schneller.
