## 1. Event-Relevanz-Scoring: Rubrik v2

**Was:** Die Bewertungs-Rubrik für Veranstaltungen wurde überarbeitet
(`lib/server/events/prompts.ts`, commit **bddd1a7** — gepusht → Vercel + Coolify,
d.h. auch der deployte deepseek-Auto-Scoring-Pfad nutzt sie jetzt).

**Vier Dimensionen, unabhängig 0.0–1.0, Gewichte (prod, DB-konfigurierbar):**
`event_score = 0.32*public_appeal + 0.32*scientific_significance + 0.21*reach + 0.15*timeliness`

- `public_appeal` = Interesse eines Laienpublikums am **Thema** (formatunabhängig)
- `scientific_significance` = Flaggschiff-/Prominenz-Charakter
- `reach` = **Format**-Öffnung (offen/groß vs. geschlossen/klein)
- `timeliness` = aktueller Anlass

**Leitlinie:** öffentliche Formate + internationale Flaggschiff-Kongresse +
tagesaktuelle Themen = HOCH; interne Formate (Fachseminare, „Monday Seminar
(Internal)", VBC-Seminare, CfP, geschlossene Graduate Schools) = gefloort.
**DE/EN-Duplikate identisch bewerten.**

**Umsetzung:** in-chat als Opus 4.8 direkt auf prod via Supabase-MCP-SQL,
Tag `llm_model = 'opus-4.8 (in-chat rubric v2)'`. Der „nicht getaggt"-Filter
macht das idempotent/resumbar. Endstand **done=171, remaining=0**.

**Top upcoming (Kontrolle):** Byzanz-Kongress 0.742 · Schrödinger-Centenary 0.706 ·
Avril-Haines/Howard-Raiffa-Lecture 0.698 · Wissenschaftsfreiheit 0.647 ·
Kinderuni 0.638. Floor: interne Montagsseminare ~0.114, „Past Events"-Platzhalter ~0.03.

### 1a. Das „Golden"-Signal / Pitch-Mechanik

Kernfix der Recalibration: `public_appeal` und `reach` waren vorher praktisch
identisch (Commit-Message: 0.99-korreliert, in 87 % der Events zahlengleich; 74 %
aller Events < 0.30). v2 entkoppelt sie bewusst, wodurch die **pa×reach-Lücke**
zum Redaktions-Signal wird:

- **GOLDEN** = pa hoch UND reach hoch → breit interessantes Thema in offenem
  Format → voll bewerben + pitchen (z. B. Kinderuni .82/.78, Byzanz .65/.85).
- **STORY-GAP** = pa hoch, reach niedrig → tolles Thema im geschlossenen Format;
  Score nur mittel, aber presse-tauglich als **Story/Interview** (z. B.
  Emily-Yeh-Geothermie .50/.30, Wahlberg-Fruchtbarkeit .55/.30). `pitch_suggestion`
  /`suggested_angle` bedienen genau diesen Fall.

**Ehrlich nachgemessen (wichtige Korrektur):** Die Kollinearität ist NICHT
aufgelöst — pa/reach korrelieren in der v2-Kohorte weiter mit **0.945** (alter
Proxy 0.982). Was sich ändert, ist die **Divergenz-Decke**: alt max |pa−re| = 0.10
und **0** Events mit Gap ≥ 0.15; v2 max **0.25** und **16** Events ≥ 0.15. Der
Story-Gap existiert also erst jetzt. Der Score-Weight-Slider wird dadurch NICHT
über pa/reach wirksam (weiter gekoppelt), sondern über die unabhängigeren Achsen
(`corr(pa,ss)=0.70`, `corr(ss,ti)=0.71`). → (a) Slider und (b) Golden-Signal
haben **verschiedene** Ursachen.

**OFFEN — TODO beim nächsten Auto-Scoring-Lauf:** Die 16 Gaps hat Opus in-chat
erzeugt, nicht deepseek über den deployten Prompt. Ob deepseek (schwächer) den
v2-Fix trägt oder auf pa≈re kollabiert, ist ungeprüft. Test: paar Events durch den
deployten deepseek-Pfad jagen, dann `corr(pa,reach)`, `max|pa−re|`, `count(gap≥0.15)`
messen. Baselines: alt 0.982/0.10/0 · Opus-v2 0.945/0.25/16.

## 3. Pooler-Egress-Fix (~1,5 GB/Tag)

**Symptom:** ~1,5 GB/Tag Shared-Pooler-Egress bei praktisch null echter Nutzung
(„nichts wird online gescored"). Diagnose via `pg_stat_statements` (66-Tage-Fenster).

**Ursachen:**
1. **Dauer-Polling der App** (~1.900 Loads/Tag auf `/` bzw. `/publications`,
   ~alle 45 s rund um die Uhr) — klar ein Healthcheck / Uptime-Monitor / Bot.
2. **`app/page.tsx` ist `force-dynamic`** → Dashboard rendert bei jedem Request neu.
3. **Score×Similarity-Scatter** schob rohe Punkte raus (bis 4.000/Load;
   kumuliert 508 Mio. Zeilen — größter Einzeltreiber).
4. **Publications-Liste** liefert je Zeile den vollen `abstract`-Text.
5. **Connection-Churn**: ~1.030 neue Verbindungen/Tag laden jeweils den pg-Typ-Katalog neu.

**Fixes — commit `a9f5f14` (gepusht, typecheck+build+494 Tests grün):**
- **`app/api/health/route.ts`** — DB-freier Liveness-Endpoint. Monitor/Coolify-
  Healthcheck hierhin statt auf `/` → Scatter, Listen-Scans UND Typ-Katalog-Reload
  entfallen pro Probe.
- **Scatter serverseitig gebinnt** (`lib/server/dashboard/fetch.ts`) — 2D-Dichte-
  Zellen `[score, similarity, count]` statt Rohpunkte. Bin-Kanten auf 40 %/85 %
  ausgerichtet → Diagnose-Quadrant-Zähler bleibt exakt. DTO
  `ScoreSimilarityPoint` → `[number,number,number]`; Chart sized Punkte per Count.
- **Listen-Projektion entschlackt** (`repos/publications.ts` + `publications/list.ts`)
  — nie gerenderte Zitations-Export-Blobs (ris/bibtex/endnote/citation_apa/
  full_text_snippet) aus dem Listen-SELECT, am DTO-Rand als `null` aufgefüllt
  (Wire-Shape unverändert).
- **Connection-Churn**: Client ist schon korrekt gepoolt (`max:1`, `prepare:false`,
  Transaction-Pooler in `db/drizzle.ts`) → Churn kommt aus Serverless-Cold-Starts,
  die der Poller auslöst → durch `/api/health` miterledigt, kein Client-Eingriff.

**⚠️ OFFENE INFRA-AKTION (nur Owner):** In **Coolify** den App-Healthcheck-Pfad
von `/` auf **`/api/health`** stellen; etwaige externe Uptime-Monitore ebenso.
Das ist der eigentliche Hebel — erst damit verschwindet der Großteil der 1,5 GB/Tag.
Gegencheck nach ein paar Tagen in Supabase → Reports → Egress (am aussagekräftigsten
nach `SELECT pg_stat_statements_reset();`).

## 4. DB-Größe (Supabase Free Tier, 500-MB-Decke)

Ausgangslage **474 MB / 500 MB** (nur ~26 MB Luft). Größte Brocken:
`publications` 299 MB (davon TOAST 153 MB), `publication_embeddings` 64 MB,
`lectures`-Subsystem ~48 MB, ~35 MB ungenutzte Indizes.

**Ausgeführt (direktes Prod-DDL, Backup davor) — 474 → 391 MB (−83 MB):**
- **Tier 1 — 10 ungenutzte Indizes gedroppt** (alle `idx_scan=0`):
  `idx_pub_original_title_trgm` (17 MB), `idx_projects_title_de_trgm`,
  `idx_pub_lead_author_trgm`, `idx_pub_keywords_gin`, `idx_persons_name_trgm`,
  + 5 Dimensions-Indizes. `DROP INDEX` gibt sofort Platz frei (kein Rewrite);
  reversibel (aus Migrations neu anlegbar), Suche/Sortierung fällt höchstens auf
  Seq-Scan (bei 38k Zeilen unkritisch).
- **Tier 2 — `TRUNCATE lectures CASCADE`** (lectures + lecture_persons/
  lecture_orgunits/project_lectures, alle jetzt 0 Zeilen). Vorträge sind reiner
  Import-Ballast: kein `/lectures`, einziger App-Read ist der Zähler auf `/upload`
  (zeigt jetzt 0 — kosmetisch). Prod-Push fasst lectures nie an → bleibt leer;
  aus lokal re-seedbar.

Headroom jetzt **~109 MB**. (Supabase-Dashboard-Anzeige samplet periodisch,
`pg_database_size()` zeigt 391 MB sofort.)

**Hinweis Schema-Drift:** Die gedroppten Indizes weichen von den Migrations-Dateien
ab. Falls ein künftiges `db push` die Migration re-appliziert, könnten sie
wiederkommen. Bei Bedarf als Migration festschreiben.

**Tier 3 — identifiziert, NICHT gemacht:** Die `publications`-TOAST (153 MB) ist
fast komplett Zitations-Export-Ballast: bibtex 30 + ris 28 + endnote 26 = **84 MB**,
citation 16 + citation_apa 18 + citation_de/en 32 = **66 MB**. Die App rendert nur
eine **geparste** Fassung eines einzelnen Zitationsfelds (CitationCard nutzt
`ParsedCitation`, nicht die Rohformate), also sind ~84–102 MB droppbar. **Aber**
echter Datei-Schrumpf braucht einen Table-Rewrite (`VACUUM FULL`/Dump-Restore),
der auf dem Free-Tier bei dieser Größe **nicht in-place** machbar ist (Peak > 500 MB).
Pfad: Spalten nullen **+** `sync-missing-pubs-to-prod` so anpassen, dass sie nie
wieder mitkommen (stoppt Nachwachsen), dann Reclaim im Wartungsfenster
(Dump→Neuaufbau oder kurz Pro-Tier). Braucht Code-Änderung + Koordination.

**Zweites Free-Tier-Limit — Egress** (~5 GB/Monat): war 2026-06-25 restringiert
(`exceed_egress_quota`, HTTP 402 auf Storage/REST; DB-Pooler lief weiter). Der
Egress-Fix (§3) zielt genau darauf.
